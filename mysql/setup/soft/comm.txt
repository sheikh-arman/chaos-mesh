## Standardized Chaos Test Suite
+
+To independently validate KubeDB MySQL Group Replication against the standardized chaos-testing methodology used in the SoftBank DBaaS evaluation ([Chaos Test Procedure](https://github.com/sheikh-arman/my-library/blob/main/chaos/chaos-testing-scripts/Chaos%20Test%20Procedure_20250806_en.md)), we ran the full **16-test suite** on a 3-node Single-Primary cluster (MySQL 8.4.8) under sustained sysbench `oltp_write_only` load.
+
+For each test we capture: the applied YAML, expected behavior, observed cluster status transitions (`Ready` / `NotReady` / `Critical`), pod role changes, sysbench throughput impact, and the final verdict.
+
+### Cluster under test
+
+| Component | Value |
+|---|---|
+| Topology | 3-node Group Replication (Single-Primary) |
+| MySQL Version | 8.4.8 |
+| Operator | KubeDB v2026.4.27 |
+| Storage | 2Gi PVC per pod (Durable, ReadWriteOnce) |
+| Memory limit | 1.5Gi per MySQL pod |
+| Chaos Engine | Chaos Mesh 2.7.2 |
+| Workload | sysbench `oltp_write_only`, 12 tables × 100k rows, 8 threads |
+| Baseline TPS | ~1,000 (single-primary write only) |
+
+### Workload preparation
+
+Before each chaos run we seed `sbtest` with 12 tables × 100k rows and start a 5–7 min `oltp_write_only` workload routed through the cluster service (which always points at the current primary):
+
+```bash
+# soft/prepare.sh
+PASS=$(kubectl get secret mysql-ha-cluster-auth -n demo -o jsonpath='{.data.password}' | base64 -d)
+SBPOD=$(kubectl get pods -n demo -l app=sysbench -o jsonpath='{.items[0].metadata.name}')
+
+kubectl exec -n demo svc/mysql-ha-cluster -c mysql -- \
+  mysql -uroot -p"$PASS" -h mysql-ha-cluster.demo \
+  -e "DROP DATABASE IF EXISTS sbtest; CREATE DATABASE sbtest;"
+
+kubectl exec -n demo $SBPOD -- sysbench oltp_write_only \
+  --mysql-host=mysql-ha-cluster --mysql-port=3306 \
+  --mysql-user=root --mysql-password="$PASS" \
+  --mysql-db=sbtest --tables=12 --table-size=100000 \
+  --threads=8 prepare
+```
+
+```bash
+# soft/load.sh
+kubectl exec -n demo $SBPOD -- sysbench oltp_write_only \
+  --mysql-host=mysql-ha-cluster --mysql-port=3306 \
+  --mysql-user=root --mysql-password="$PASS" \
+  --mysql-db=sbtest --tables=12 --table-size=100000 \
+  --threads=8 --time=300 --report-interval=10 \
+  --mysql-ignore-errors=2013,1290,1213,1205 run
+```
+
+### Test 01 — Pod Failure on Primary (5 min)
+
+Simulate a hung container by injecting a `pod-failure` fault on the current primary pod for 5 minutes.
+
+```yaml
+apiVersion: chaos-mesh.org/v1alpha1
+kind: PodChaos
+metadata:
+  name: test-kubedb-primary-pod-failure
+  namespace: demo
+spec:
+  selector:
+    namespaces:
+      - demo
+    labelSelectors:
+      app.kubernetes.io/component: database
+      app.kubernetes.io/managed-by: kubedb.com
+      app.kubernetes.io/name: mysqls.kubedb.com
+      kubedb.com/role: primary
+  mode: all
+  action: pod-failure
+  duration: 5m
+```
+
+```bash
+➤ kubectl apply -f soft/01-pod-failure-primary.yaml
+podchaos.chaos-mesh.org/test-kubedb-primary-pod-failure created
+```
+
+**Expected behavior:**
+Primary container becomes unavailable → cluster transitions `Ready` → `NotReady` → after Group Replication elects a new primary, state moves to `Critical` (one replica missing). When the 5-min fault expires, the old primary container restarts, rejoins as `SECONDARY`, and the cluster returns to `Ready`. No data loss expected.
+
+**Actual result — observed timeline:**
+
+| Wall-clock | Δ from chaos | Event | DB Status |
+|---|---|---|---|
+| 11:39:00 | — | Pre-chaos baseline (pod-2 = primary, all 3 ONLINE) | `Ready` |
+| 11:39:21 | 0s | `PodChaos` applied (pod-2 targeted) | `Ready` |
+| 11:39:26 | +5s | Primary unreachable detected | **`NotReady`** |
+| 11:39:29 | +8s | Group Replication promotes **pod-1 → PRIMARY** | `NotReady` |
+| 11:39:42 | +21s | Operator marks pod-2 unhealthy | **`Critical`** |
+| 11:44:21 | +5m00s | Chaos auto-recovered, pod-2 container restarts | `Critical` |
+| 11:45:40 | +6m19s | pod-2 in `RECOVERING` (catching up via incremental clone) | `Critical` |
+| ~11:48–52 | ~+9–13m | pod-2 reaches `ONLINE` `SECONDARY` | **`Ready`** |
+
+```bash
+# Final state — all 3 ONLINE, data intact
+➤ kubectl get mysql -n demo
+NAME               VERSION   STATUS   AGE
+mysql-ha-cluster   8.4.8     Ready    16h
+
+➤ kubectl get pods -n demo -l app.kubernetes.io/name=mysqls.kubedb.com -L kubedb.com/role
+NAME                 READY   STATUS    ROLE
+mysql-ha-cluster-0   2/2     Running   standby
+mysql-ha-cluster-1   2/2     Running   primary    # promoted from standby
+mysql-ha-cluster-2   2/2     Running   standby    # rejoined after chaos
+
+➤ SELECT MEMBER_HOST, MEMBER_STATE, MEMBER_ROLE FROM performance_schema.replication_group_members;
+mysql-ha-cluster-2.…  ONLINE  SECONDARY
+mysql-ha-cluster-1.…  ONLINE  PRIMARY
+mysql-ha-cluster-0.…  ONLINE  SECONDARY
+
+➤ SELECT COUNT(*) FROM sbtest.sbtest1;   # 100000 (unchanged)
+```
+
+**sysbench impact:** initial connection lost (error 2013) within 18 s; after reconnect, TPS dipped 1012 → 314 → recovered to ~1000 within 60 s on the new primary.
+
+**Result: PASS** — failover completed in **8 s**, full cluster recovery in ~9–13 min after fault clearance. Zero data loss, zero errant GTIDs.

signal.txt is the coordinator's instruction file to the init script (run.sh / run_innodb.sh). Possible
  values: join_in_cluster, clone, reboot_from_completeOutage, etc. Each operation writes the signal,
  restarts mysqld, the init script re-reads the signal on boot and acts on it. The init script is
  supposed to delete signal.txt once it's processed. If the file is still there, either:

  1. The init script processed it but didn't unlink it (bug in run.sh / run_innodb.sh — search for rm
  signal.txt).
  2. The init script crashed/exited before unlinking.
  3. Coordinator keeps re-writing it because it still detects the same condition.

  To debug, check what's inside signal.txt:

  cat /scripts/signal.txt

  That tells you which operation the coordinator is asking for. Then look at the latest coordinator log
  lines to see why it thinks that operation is needed:

  kubectl logs <pod> -c mysql-coordinator --tail=30 | grep -i "creating
  .*signal\|reboot\|clone\|join_in_cluster"

  Most common causes of a stuck signal:
  - join_in_cluster — coordinator thinks pod isn't in GR, but pod is actually in GR (race or label
  mismatch). Init script's is_already_in_cluster() check returns true → skips re-join → doesn't clear
  signal.
  - clone — pod has errant GTIDs; coordinator wants clone but blocks on /scripts/approve-clone (the same
  gate from the OOM chaos test we did earlier).
  - reboot_from_completeOutage — coordinator thinks all members are down; one of them rejects reboot
  because GR thinks it's still ONLINE.

  Also note ack-from-0 and ack-from-2 (4 bytes each, no ack-from-1). Those are the inter-pod ack markers
  used by the elect-primary / reboot flow. Missing ack-from-1 suggests pod-1 hasn't acked. If this is
  pod-1 itself looking at its own scripts dir, that's expected — it doesn't write its own ack. But if
  this is pod-0 or pod-2, missing ack-from-1 means pod-1 isn't responding to whatever the coordination
  round is.

  stuck?
